# AMC10 CoT Fine-Tuning with LoRA

Fine-tuning **Qwen2.5-1.5B-Instruct** on synthetically-generated chain-of-thought reasoning traces for AMC10 math competition problems, using LoRA (low-rank adaptation).

---

## Motivation

Full fine-tuning is compute-heavy and out of reach for consumer GPUs. LoRA sidesteps that by training two small low-rank matrices per layer instead of the full weight matrix — a fraction of the original cost. This project applies LoRA to specialize Qwen2.5-1.5B-Instruct on AMC10 problems using CoT traces generated by OpenAI's o3-mini as a teacher model.

---

## Pipeline

1. **Scrape** — `scraper.py` pulls AMC10 problems, answer choices, and community solutions from AoPS (aops.com) using Selenium with one fresh Chrome session per page load to avoid Cloudflare rate-limiting.
2. **Synthesize** — o3-mini generates a step-by-step reasoning chain for each problem/solution pair. These become the training targets.
3. **Train** — `training.py` fine-tunes Qwen2.5-1.5B-Instruct via LoRA on the reasoning traces using SFTTrainer with early stopping.
4. **Benchmark** — `benchmark.py` runs both the base model and the fine-tuned adapter on the held-out test set, then uses o3-mini as a grading judge to evaluate final answers by difficulty tier.

## Dataset

- **Source:** Art of Problem Solving wiki (AMC10A/B, multiple years)
- **Total problems:** 979 (after filtering token-length outliers)
- **Split:** 686 train / 97 eval / 196 test (~70/10/20)
- **Each record:** problem text, answer choices, correct answer letter, community solution, o3-mini reasoning chain

---

## LoRA Configuration

| Hyperparameter | Value |
|---|---|
| Rank (`r`) | 16 |
| Alpha | 32 |
| Dropout | 0.06 |
| Target modules | `q_proj`, `k_proj`, `v_proj`, `o_proj`, `gate_proj`, `up_proj`, `down_proj` |
| Learning rate | 1e-4 |
| Effective batch size | 8 (batch 1 × grad accum 8) |
| Max epochs | 4 (early stopping patience = 3) |
| Truncation length | 1500 tokens |

Targeting both attention projections and the MLP block (gate/up/down) — attention-only LoRA tends to restyle output rather than shift computation.

---

## Results

Despite multiple training runs and hyperparameter adjustments, the fine-tuned model consistently broke even with or slightly underperformed the base model on the test set.

Inspecting the outputs revealed the core issue: the model absorbed o3-mini's **tone** (first-person phrasing like "I start by noticing…") rather than its reasoning strategy. The fine-tuned model mimicked the surface style of the teacher without gaining actual problem-solving capability.

---

## Conclusion

Many will point out that the problem (copying tone rather than reasoning ability) could maybe be solved with simply modifying the system prompt of o3-mini. But that doesn't address the root of the problem. CoT is simply not sufficient for training models on complex tasks, like in this case, solving AMC problems. This is not simply a personally-investigative claim, but rather a factual and proven statement addressed in several professional papers. Models learn "what," not "how." In simpler environments, it's possible that CoT would prove itself to be competent. But in this case, it just doesn't work.

## Quick Start

```bash
pip install torch transformers trl peft bitsandbytes datasets openai selenium matplotlib

# 1. Scrape problems (requires ChromeDriver)
python scraper.py --years 2015-2025 --contest both

# 2. Generate CoT reasoning traces (requires OPENAI_API_KEY)
python reasoning.py

# 3. Fine-tune
python training.py

# 4. Benchmark base vs. fine-tuned
python benchmark.py
```

> **Note:** Training requires a CUDA GPU. An `OPENAI_API_KEY` is needed for reasoning trace generation and benchmark judging.
